# Codebase Documentation for PingulaCore-Service

This document provides a detailed overview of the PingulaCore-Service codebase, covering its endpoints, agents, functions, database structure, and pipeline flows.

---

## 1. Endpoints

The API endpoints are defined using FastAPI and are primarily located in `app/api/agent.py` and `app/api/pipeline.py`.

### Endpoints from `app/api/agent.py`

These endpoints primarily interact with the AI agents for various content generation and validation tasks.

*   **POST `/agents/main/generate-question/run`**
    *   **Description:** Initiates the agent responsible for generating questions from YAML input.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/main/generate-layout/run`**
    *   **Description:** Initiates the agent responsible for generating layout plans from a question specification.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/main/generate-html/run`**
    *   **Description:** Initiates the agent responsible for generating HTML content from a layout plan and question specification.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/validation/extract-rules/run`**
    *   **Description:** Initiates the agent responsible for extracting validation rules from YAML input.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/validation/evaluate-rule/run`**
    *   **Description:** Initiates the agent responsible for evaluating a specific validation rule against a question specification.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/validation/validate-question-layout/run`**
    *   **Description:** Initiates the agent responsible for validating the consistency between a question specification and a layout plan.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/validation/validate-layout-html/run`**
    *   **Description:** Initiates the agent responsible for validating the visual quality of generated HTML from a layout.
    *   **Response Model:** `StandaloneAgentResponse`

*   **POST `/agents/helper/generate-composite-image/run`**
    *   **Description:** Initiates the agent responsible for generating composite images.
    *   **Response Model:** `StandaloneAgentResponse`

*   **GET `/agent-runs/{agent_name}/{run_id}`**
    *   **Description:** Retrieves the status or detailed information of a specific agent run.
    *   **Parameters:**
        *   `agent_name` (path): The name of the agent (e.g., `main_generate_question`).
        *   `run_id` (path): The unique identifier of the agent run.
    *   **Response Model:** `AgentRunGetResponse`

### Endpoints from `app/api/pipeline.py`

These endpoints manage the overall content generation pipelines and provide access to intermediate results and runtime information.

*   **POST `/pipelines/full/run`**
    *   **Description:** Executes a complete content generation pipeline, from YAML input to final HTML and rendered image.
    *   **Response Model:** `FullPipelineRunResponse`

*   **GET `/runtime-info`**
    *   **Description:** Provides various runtime information about the application.
    *   **Response Model:** `RuntimeInfoResponse`

*   **GET `/yaml-files`**
    *   **Description:** Lists available YAML configuration files.
    *   **Response Model:** `YamlFilesResponse`

*   **GET `/yaml-files/{filename}`**
    *   **Description:** Retrieves the content of a specific YAML file.
    *   **Parameters:**
        *   `filename` (path): The name of the YAML file.
    *   **Response Model:** `YamlFileContentResponse`

*   **GET `/sp-files/q_json`**
    *   **Description:** Lists available `QuestionSpec` JSON files generated by sub-pipelines.
    *   **Response Model:** `SpFilesResponse`

*   **GET `/sp-files/layout`**
    *   **Description:** Lists available `LayoutPlan` JSON files generated by sub-pipelines.
    *   **Response Model:** `SpFilesResponse`

*   **GET `/sp-files/q_html`**
    *   **Description:** Lists available HTML files generated by sub-pipelines.
    *   **Response Model:** `SpFilesResponse`

*   **GET `/sp-files/q_json/{filename}`**
    *   **Description:** Retrieves the content of a specific `QuestionSpec` JSON file.
    *   **Parameters:**
        *   `filename` (path): The name of the JSON file.
    *   **Response Model:** `SpJsonFileResponse`

*   **GET `/sp-files/layout/{filename}`**
    *   **Description:** Retrieves the content of a specific `LayoutPlan` JSON file.
    *   **Parameters:**
        *   `filename` (path): The name of the JSON file.
    *   **Response Model:** `SpJsonFileResponse`

*   **GET `/sp-files/q_html/{filename}`**
    *   **Description:** Retrieves the content of a specific HTML file.
    *   **Parameters:**
        *   `filename` (path): The name of the HTML file.
    *   **Response Model:** `SpHtmlFileResponse`

*   **GET `/assets/{filename:path}`**
    *   **Description:** Serves static asset files (e.g., images) from the `catalog/` and `generated_assets/` directories.
    *   **Parameters:**
        *   `filename` (path): The full path to the asset within the asset directories.

*   **POST `/pipelines/sub/yaml-to-question/run`**
    *   **Description:** Runs the "YAML to Question" sub-pipeline independently.
    *   **Response Model:** `YamlToQuestionRunResponse`

*   **POST `/pipelines/sub/question-to-layout/run`**
    *   **Description:** Runs the "Question to Layout" sub-pipeline independently.
    *   **Response Model:** `QuestionToLayoutRunResponse`

*   **POST `/pipelines/sub/layout-to-html/run`**
    *   **Description:** Runs the "Layout to HTML" sub-pipeline independently.
    *   **Response Model:** `LayoutToHtmlRunResponse`

*   **GET `/pipelines/{pipeline_id}`**
    *   **Description:** Retrieves detailed information about a specific full pipeline run.
    *   **Parameters:**
        *   `pipeline_id` (path): The unique identifier of the pipeline run.
    *   **Response Model:** `PipelineGetResponse`

*   **GET `/pipelines/{pipeline_id}/agent-runs`**
    *   **Description:** Lists all agent runs associated with a specific full pipeline run.
    *   **Parameters:**
        *   `pipeline_id` (path): The unique identifier of the pipeline run.
    *   **Response Model:** `list[PipelineAgentLinkResponse]`

*   **GET `/pipelines/{pipeline_id}/logs`**
    *   **Description:** Retrieves all log entries for a specific full pipeline run.
    *   **Parameters:**
        *   `pipeline_id` (path): The unique identifier of the pipeline run.
    *   **Response Model:** `list[PipelineLogEntryResponse]`

*   **GET `/sub-pipelines/{sub_pipeline_id}`**
    *   **Description:** Retrieves detailed information about a specific sub-pipeline run.
    *   **Parameters:**
        *   `sub_pipeline_id` (path): The unique identifier of the sub-pipeline run.
    *   **Response Model:** `SubPipelineGetResponse`

*   **GET `/sub-pipelines/{sub_pipeline_id}/agent-runs`**
    *   **Description:** Lists all agent runs associated with a specific sub-pipeline run.
    *   **Parameters:**
        *   `sub_pipeline_id` (path): The unique identifier of the sub-pipeline run.
    *   **Response Model:** `list[PipelineAgentLinkResponse]`

*   **GET `/sub-pipelines/{sub_pipeline_id}/logs`**
    *   **Description:** Retrieves all log entries for a specific sub-pipeline run.
    *   **Parameters:**
        *   `sub_pipeline_id` (path): The unique identifier of the sub-pipeline run.
    *   **Response Model:** `list[PipelineLogEntryResponse]`

## 2. Agents

The core AI agents are managed and orchestrated by the `AgentService` class, defined in `app/agents/agent_service.py`. This service acts as a centralized interface for running various AI models, handling aspects like model selection, retries, and error management.

### `AgentService` Class (`app/agents/agent_service.py`)

*   **Purpose:** Orchestrates the execution of different AI agents. It provides a unified way to interact with various models, including handling model fallbacks, retry logic for transient errors, and managing execution in sync/async contexts.
*   **Key Method: `_run_agent(...)`:**
    *   This is the internal method that executes an AI agent. It takes parameters such as `model_name`, expected `output_type`, `system_prompt` (instructions for the AI), the `payload` (input data), `agent_name`, and `thinking_level`. It incorporates retry mechanisms and model fallback strategies.
*   **Stub Agents:** The service includes "stub" implementations for all agents. These are non-AI versions that return mock data, useful for local development, testing, or when AI API keys are unavailable. They are activated when `self.settings.use_stub_agents` is `True` or if an AI agent call fails.

### Specific Agent Functions

The `AgentService` exposes methods for each type of AI agent, corresponding to the API endpoints and pipeline stages. Each method typically calls `_run_agent` with specific instructions and output types.

1.  **`generate_question(yaml_content, feedback)`**
    *   **Role:** Generates a structured `QuestionSpec` object from raw YAML curriculum content.
    *   **Instructions (`QUESTION_AGENT_INSTRUCTIONS`):** Focuses on generating only the `QuestionSpec` without layout or rendering details.
    *   **Input:** YAML content (dict), optional feedback (str).
    *   **Output:** `QuestionSpec`.

2.  **`extract_rules(yaml_content)`**
    *   **Role:** Extracts atomic validation rules from YAML input, prioritizing critical constraints.
    *   **Instructions (`RULE_EXTRACTOR_INSTRUCTIONS`):** Aims for concise, testable rules, merging similar ones, with a maximum of 12 rules.
    *   **Input:** YAML content (dict).
    *   **Output:** `RuleExtractionResult` (containing `ValidationRule` objects).

3.  **`evaluate_rule(rule, question)`**
    *   **Role:** Evaluates a single `ValidationRule` against a generated `QuestionSpec`.
    *   **Instructions (`RULE_EVALUATOR_INSTRUCTIONS`):** Determines if the rule passes, partially passes, or fails.
    *   **Input:** `ValidationRule`, `QuestionSpec`.
    *   **Output:** `RuleEvaluation`.

4.  **`evaluate_rules_parallel(rules, question, parallelism, on_progress)`**
    *   **Role:** Concurrently evaluates multiple validation rules, providing progress updates.
    *   **Input:** List of `ValidationRule`s, `QuestionSpec`, `parallelism` level, and an optional progress callback.
    *   **Output:** `RuleEvaluationSet` (a collection of `RuleEvaluation` results).

5.  **`generate_layout(question, feedback)`**
    *   **Role:** Creates a `LayoutPlan` from a `QuestionSpec`, detailing asset placement and layering.
    *   **Instructions (`LAYOUT_PLANNER_INSTRUCTIONS`):** Emphasizes proper placement of opaque AI assets and transparent catalog assets, respecting layering (`z_index`) for critical elements.
    *   **Input:** `QuestionSpec`, optional feedback (str).
    *   **Output:** `LayoutPlan`.

6.  **`validate_question_layout(question, layout)`**
    *   **Role:** Validates the consistency and correctness of a `LayoutPlan` relative to its `QuestionSpec`.
    *   **Instructions (`LAYOUT_VALIDATOR_INSTRUCTIONS`):** Checks for multi-scene coverage and ensures opaque assets don't hide critical foreground elements.
    *   **Input:** `QuestionSpec`, `LayoutPlan`.
    *   **Output:** `QuestionLayoutValidationResult`.

7.  **`generate_html(question, layout, asset_map, feedback)`**
    *   **Role:** Produces HTML content based on a `QuestionSpec` and `LayoutPlan`, incorporating an asset map for image paths.
    *   **Instructions (`HTML_GENERATOR_INSTRUCTIONS`):** Focuses on maintaining pedagogical intent and using correct asset `src` values.
    *   **Input:** `QuestionSpec`, `LayoutPlan`, `asset_map` (dict), optional feedback (str).
    *   **Output:** `GeneratedHtml`.

8.  **`render_html_to_image(html_content, asset_map, question_id)`**
    *   **Role:** Renders generated HTML into a screenshot image using a headless browser (e.g., Chrome).
    *   **Input:** HTML content (str), optional asset map (dict), optional question ID (str).
    *   **Output:** Path to the generated image (str).

9.  **`validate_html(html_content, rendered_image_path)`**
    *   **Role:** Visually evaluates the quality of the final rendered HTML image.
    *   **Instructions (`HTML_VALIDATOR_INSTRUCTIONS`):** Assesses readability, spacing, alignment, occlusion, option clarity, and overall pedagogical usability. Provides actionable feedback for HTML editing.
    *   **Input:** HTML content (str), path to the rendered image (str).
    *   **Output:** `HtmlValidationResult`.

10. **`generate_composite_image(asset, max_retries, catalog_context_filenames)`**
    *   **Role:** Generates an image using an AI image model, potentially incorporating existing catalog images as stylistic and contextual references.
    *   **Instructions (`IMAGE_COMPOSITE_SYSTEM_INSTRUCTIONS`):** Guides the image generation to ensure style, perspective, lighting, scale, and color harmony with catalog elements, and to leave usable empty space.
    *   **Input:** `AssetSpec`, maximum retries (int), optional list of catalog filenames for context.
    *   **Output:** `CompositeImageResult`.

## 3. Database

The database component uses SQLAlchemy ORM to define models, manage connections, and handle data persistence.

### Schema Definition (`app/db/models.py`)

The `app/db/models.py` file defines the SQLAlchemy ORM models, mapping Python classes to database tables.

*   **`Pipeline` (Table: `pipelines`)**:
    *   Represents a full pipeline run.
    *   **Fields:** `id` (PK), `mode`, `yaml_filename`, `status`, `retry_config_json`, `error`, `created_at`, `finished_at`.
    *   **Relationships:** One-to-many with `SubPipeline` (`sub_pipelines`).

*   **`SubPipeline` (Table: `sub_pipelines`)**:
    *   Represents an individual stage within a pipeline (e.g., YAML to Question).
    *   **Fields:** `id` (PK), `pipeline_id` (FK to `pipelines`), `mode`, `kind`, `status`, `input_json`, `output_json`, `error`, `created_at`, `finished_at`.
    *   **Relationships:** Many-to-one with `Pipeline`.

*   **`PipelineAgentLink` (Table: `pipeline_agent_links`)**:
    *   Links agent runs to their parent pipeline or sub-pipeline.
    *   **Fields:** `id` (PK, autoincrement), `pipeline_id` (FK), `sub_pipeline_id` (FK), `agent_name`, `agent_table`, `agent_run_id`, `created_at`.

*   **`PipelineLog` (Table: `pipeline_logs`)**:
    *   Stores log entries for pipeline and sub-pipeline activities.
    *   **Fields:** `id` (PK, autoincrement), `pipeline_id` (FK), `sub_pipeline_id` (FK), `mode`, `level`, `component`, `message`, `details_json`, `created_at`.

*   **`AgentRunMixin`**:
    *   A mixin class defining common fields for all agent-specific run tables. Not a table itself.
    *   **Fields:** `id` (PK), `mode`, `pipeline_id` (FK), `sub_pipeline_id` (FK), `attempt_no`, `status`, `input_json`, `output_json`, `feedback_text`, `error`, `model_name`, `question_id`, `schema_version`, `started_at`, `finished_at`.

*   **Agent-Specific Run Tables**:
    *   Inherit from `AgentRunMixin` and `Base`. Examples include `AgentMainQuestionRun`, `AgentMainLayoutRun`, `AgentRuleExtractionRun`, etc. Each corresponds to a specific agent's execution history.
    *   **Purpose:** To store detailed records of each agent's invocation, inputs, outputs, and status.

### Repository (`app/db/repository.py`)

The `app/db/repository.py` file provides a set of functions for performing CRUD (Create, Read, Update, Delete) operations on the database models. It abstracts the direct SQLAlchemy interactions, offering a clean interface for data access.

*   **`create_pipeline`, `finish_pipeline`**: For managing `Pipeline` records.
*   **`create_sub_pipeline`, `finish_sub_pipeline`**: For managing `SubPipeline` records.
*   **`record_agent_run`**: Creates a record in the appropriate agent-specific run table and establishes a `PipelineAgentLink`.
*   **`record_pipeline_log`**: Persists log messages to the `pipeline_logs` table.
*   **`list_pipeline_logs`, `list_sub_pipeline_logs`**: Retrieve log entries.
*   **`get_pipeline`, `get_sub_pipeline`, `get_agent_run`**: Retrieve specific records by ID.
*   **`list_pipeline_links`, `list_sub_pipeline_links`**: Retrieve agent links.
*   **Helper Functions:** `_to_json_text` (for serializing JSON), `_extract_question_meta` (for parsing question metadata).

### Database Connection (`app/db/database.py`)

The `app/db/database.py` file handles the setup of the SQLAlchemy engine, session management, and database initialization.

*   **`Base`**: The declarative base for SQLAlchemy models.
*   **`engine`**: The SQLAlchemy engine, configured using `DATABASE_URL` from application settings.
*   **`SessionLocal`**: A session maker configured for database interactions.
*   **`get_db()`**: A dependency injection function that yields a database session and ensures its proper closure.
*   **`init_db()`**: Creates all tables defined in `app/db/models.py` in the database.

## 4. Pipeline Flows

The core logic for orchestrating the content generation process is encapsulated within the `PipelineService` in `app/services/pipeline_service.py`. This service defines and executes various pipeline stages, manages retries, and records all activities.

### `PipelineService` Class (`app/services/pipeline_service.py`)

*   **Purpose:** Manages the entire content generation workflow, from initial YAML input to validated HTML output. It coordinates calls to AI agents, handles retry logic, logs pipeline progress, and persists intermediate and final results to both the database and the filesystem.
*   **Dependencies:**
    *   `db: Session`: For database operations (logging, status updates).
    *   `settings: Settings`: Application configuration.
    *   `agents: AgentService`: The interface to AI agents.
*   **Logging:** Uses `_log` helper method, which calls `app.services.pipeline_log_service.write_pipeline_log`, to record detailed progress and status messages to the database and standard output.
*   **Persistence:** Uses `_persist_sub_output_*` methods (which internally call `app.services.sub_pipeline_files_service`) to save `QuestionSpec`, `LayoutPlan`, and generated HTML to designated directories (`sp_files/q_json`, `sp_files/layout`, `sp_files/q_html`).

### Major Pipeline Stages (Implemented as Internal Loops with Retries)

1.  **`_run_yaml_to_question_loop` (YAML -> Question)**
    *   **Input:** Raw YAML content.
    *   **Steps:**
        *   **Rule Extraction:** `agents.extract_rules` processes the YAML to identify validation rules.
        *   **Iterative Question Generation & Validation:**
            *   `agents.generate_question` creates a `QuestionSpec`.
            *   `agents.evaluate_rules_parallel` validates the generated question against the extracted rules.
            *   If validation fails, feedback is generated from failed rules and fed back into the next `generate_question` attempt. This loop continues until the question passes validation or retry limits are met.
    *   **Output:** A valid `QuestionSpec`, rule evaluation results, and the number of generation attempts.

2.  **`_run_question_to_layout_loop` (Question -> Layout)**
    *   **Input:** A validated `QuestionSpec`.
    *   **Steps:**
        *   **Iterative Layout Generation & Validation:**
            *   `agents.generate_layout` creates a `LayoutPlan` based on the `QuestionSpec`.
            *   `agents.validate_question_layout` checks the `LayoutPlan` for consistency and correctness.
            *   If validation fails, feedback is used to guide the next `generate_layout` attempt. This loop continues until a valid layout is produced or retry limits are met.
    *   **Output:** A valid `LayoutPlan`, layout validation results, and the number of generation attempts.

3.  **`_run_layout_to_html_loop` (Layout -> HTML)**
    *   **Input:** A validated `QuestionSpec` and `LayoutPlan`.
    *   **Steps:**
        *   **Asset Preparation:** `agents.generate_composite_image` generates AI-created images, and `asset_map` is built to resolve paths for all assets (including existing catalog images).
        *   **Iterative HTML Generation & Visual Validation:**
            *   `agents.generate_html` produces HTML content using the `QuestionSpec`, `LayoutPlan`, and `asset_map`.
            *   `agents.render_html_to_image` takes a screenshot of the generated HTML.
            *   `agents.validate_html` visually evaluates the rendered image and HTML for quality.
            *   If validation fails, feedback guides the next `generate_html` attempt. This loop continues until the HTML is visually acceptable or retry limits are exhausted.
    *   **Output:** Generated HTML content, visual validation results, asset map, and the path to the final rendered image.

### Full Pipeline Execution (`run_full_pipeline`)

*   **Purpose:** Orchestrates the sequential execution of the three major pipeline stages.
*   **Workflow:**
    1.  Records a new `Pipeline` entry in the database.
    2.  Creates `SubPipeline` entries for each of the three stages.
    3.  Loads the initial YAML content using `app.services.yaml_service.load_yaml_file`.
    4.  Executes `_run_yaml_to_question_loop`, `_run_question_to_layout_loop`, and `_run_layout_to_html_loop` in sequence.
    5.  Updates the status and persists outputs for each `SubPipeline` and the main `Pipeline` in the database and filesystem.
    6.  Includes comprehensive error handling and logging at each step.

### Independent Sub-pipeline Execution

The `PipelineService` also exposes methods (`run_sub_yaml_to_question`, `run_sub_question_to_layout`, `run_sub_layout_to_html`) to run each of the three sub-pipelines independently. These methods follow a similar logic to their internal loop counterparts but manage their own `SubPipeline` record in the database.

## 5. Core Services and Functions

This section covers foundational modules that provide general utilities and configuration for the application.

### Configuration (`app/core/config.py`)

*   **Purpose:** Manages application-wide settings, loading values from environment variables or providing sensible defaults. It ensures a consistent configuration across the application.
*   **`Settings` Dataclass:** Defines all configurable parameters (paths, AI model names, retry counts, parallelism, feature flags like `use_stub_agents`).
*   **`_load_dotenv_file()`:** A utility to load environment variables from a `.env` file without overwriting existing system environment variables.
*   **`build_settings()`:** Initializes the `Settings` object, resolving paths and determining default AI models based on available API keys.
*   **`get_settings()`:** A cached function (`@lru_cache`) that provides a singleton instance of the `Settings` object, ensuring settings are loaded only once and consistently accessed.

### Pipeline Logging (`app/services/pipeline_log_service.py`)

*   **Purpose:** Provides a dedicated function for logging pipeline-related events.
*   **`write_pipeline_log()`:** Writes log messages to both standard output (for real-time monitoring) and persists them to the `pipeline_logs` table in the database via the `repository`.

### Retry Configuration (`app/services/retry_service.py`)

*   **Purpose:** Defines and manages retry parameters for the various pipeline stages.
*   **`RetrySettings` Dataclass:** Stores maximum retry counts and parallelism settings for question generation, layout generation, HTML generation, image generation, and rule evaluation.
*   **`merge_retry_config()`:** A function that merges user-provided retry configurations with default application settings, allowing for flexible and granular control over retry behavior.

### Sub-pipeline File Persistence (`app/services/sub_pipeline_files_service.py`)

*   **Purpose:** Manages the saving and loading of intermediate and final outputs of sub-pipelines (Question JSON, Layout JSON, HTML) to the filesystem.
*   **File Structure:** Organizes files under the `sp_files/` directory, with subdirectories for `q_json/`, `layout/`, and `q_html/`.
*   **Functions:** Provides methods like `write_question_file`, `write_layout_file`, `write_html_file` for saving, and `read_json_file`, `read_html_file`, `list_files` for retrieval. File naming incorporates timestamps and IDs for traceability.

### YAML Service (`app/services/yaml_service.py`)

*   **Purpose:** Handles the location and loading of YAML configuration files, which are the initial input for the content generation pipelines.
*   **`resolve_yaml_path()`:** Locates YAML files by checking a primary and a fallback directory.
*   **`load_yaml_file()`:** Reads and safely parses YAML file content into a Python dictionary, ensuring the top-level structure is a dictionary.

## 6. Frontend

The frontend is a modern web application built using **React**, **TypeScript**, and **Vite**, with **React Router DOM** for client-side routing. It's designed to provide a user interface for interacting with the backend services.

### Technologies Used

*   **React (v19.x):** A JavaScript library for building user interfaces.
*   **TypeScript (v6.0.x):** A superset of JavaScript that adds static types, improving code quality and maintainability.
*   **Vite (v8.0.x):** A fast build tool that provides a rapid development experience for modern web projects.
*   **React Router DOM (v7.x):** Used for declarative routing within the single-page application.
*   **ESLint:** For static code analysis to enforce code style and identify potential issues.
*   **Vitest & React Testing Library:** For unit and integration testing of React components.
*   **JSDOM:** Provides a simulated DOM environment for running frontend tests without a browser.

### Project Structure (`frontend/src/`)

The `src/` directory follows a common and organized structure for React applications:

*   **`main.tsx`:** The application's entry point. It bootstraps the React application, rendering the root `<App />` component within `<StrictMode>` and `<BrowserRouter>`.
*   **`App.tsx`:** The main component of the application, likely containing the primary layout and route definitions.
*   **`index.css` & `App.css`:** CSS files for global styles and application-specific styles, respectively.
*   **`types.ts`:** Dedicated file for global TypeScript type definitions, enhancing type safety across the project.
*   **`assets/`:** Contains static assets such as images, icons, or fonts used by the application.
*   **`components/`:** A collection of reusable UI components (e.g., buttons, cards, forms) that can be composed to build complex interfaces.
*   **`hooks/`:** Custom React Hooks to encapsulate reusable stateful logic and side effects.
*   **`lib/`:** Utility functions, helper modules, or third-party library configurations.
*   **`pages/`:** Components that represent distinct views or "pages" of the application, typically corresponding to different routes defined by `react-router-dom`.
*   **`test/`:** Contains additional test-related files, configurations, or mocks, complementing the component-level tests.

### Development and Build Process

*   **Development Server:** `npm run dev` (or `pnpm dev`, `yarn dev`) starts a development server powered by Vite, offering hot module replacement and fast feedback loops.
*   **Build:** `npm run build` compiles the TypeScript code and bundles the application for production deployment.
*   **Linting:** `npm run lint` runs ESLint to check for code quality and style consistency.
*   **Testing:** `npm run test` executes unit and integration tests using Vitest, often leveraging React Testing Library to simulate user interactions and assert UI behavior. `npm run test:watch` runs tests in watch mode.
